# IPC Protocol Specification: C++/TypeScript Boundary Contract

**Version**: 1.0  
**Status**: Proposed  
**Date**: 2026-04-10  
**Authors**: Architecture Team

## 1. Overview

This document defines the formal communication contract between the TypeScript execution engine and a future C++ execution core. Both implementations must satisfy this spec. A C++ developer can implement against this document without access to the TypeScript codebase.

### 1.1 Design Principles

1. **Capital safety first** -- every failure mode must be safe (close positions, not open new ones)
2. **Zero-copy where it matters** -- market data via shared memory ring buffers; commands via UDS
3. **Incremental migration** -- TypeScript and C++ coexist; exchanges migrate one at a time
4. **Observable** -- every message carries timing metadata for latency analysis
5. **Deterministic** -- state machine transitions are identical in both implementations

### 1.2 Transport Summary

| Channel | Transport | Encoding | Direction | Use Case |
|---------|-----------|----------|-----------|----------|
| Commands | Unix Domain Socket | FlatBuffers | API -> Engine | Position lifecycle, engine control |
| Responses | Unix Domain Socket | FlatBuffers | Engine -> API | Command results |
| Events | Unix Domain Socket | FlatBuffers | Engine -> API | State changes, fills, heartbeats |
| Market Data | Shared Memory Ring Buffer | Raw binary (64-byte slots) | Writer -> Reader | Price ticks, BBO updates |
| Low-freq Market | Unix Domain Socket | FlatBuffers | Engine -> API | Funding rates, L2 snapshots |

## 2. Unix Domain Socket Protocol

### 2.1 Wire Format

Every UDS message uses length-prefixed FlatBuffers framing:

```
+-------------------+----------------------------+
| Length (4 bytes)   | FlatBuffers Payload        |
| uint32 big-endian  | (Envelope table)           |
+-------------------+----------------------------+
```

- Maximum payload size: 1 MB (1,048,576 bytes)
- Byte order: FlatBuffers uses little-endian internally; the 4-byte length prefix is big-endian (matching the existing Phase 1 JSON protocol for backwards compatibility)

### 2.2 Migration from JSON to FlatBuffers

During transition, both sides must detect the encoding:

- **Phase 1 (current)**: JSON payloads inside the length-prefixed frame
- **Phase 2 (this spec)**: FlatBuffers payloads

Detection: the first 4 bytes of a FlatBuffers buffer are a uint32 offset to the root table, which is always >= 4. JSON payloads start with `{` (0x7B) or `[` (0x5B). The receiver checks `payload[0]`: if < 0x20, treat as FlatBuffers; otherwise treat as JSON.

### 2.3 Message Envelope

Every message across UDS is wrapped in an `Envelope` (see `schemas/envelope.fbs`):

| Field | Type | Description |
|-------|------|-------------|
| `seq` | uint64 | Monotonically increasing per-sender sequence number |
| `timestamp_ns` | uint64 | Nanosecond monotonic clock timestamp |
| `correlation_id` | [ubyte] (16 bytes) | UUID v4 as raw bytes |
| `msg_type` | MessageType enum | Discriminator for payload type |
| Payload fields | various | Exactly one payload table is populated |

### 2.4 Sequence Number Contract

- Each sender (API layer, Engine) maintains its own independent sequence counter starting at 0
- Sequence numbers are strictly monotonic (gaps indicate lost messages)
- The receiver tracks the last seen sequence from each sender
- If a gap is detected, the receiver logs a warning but does NOT reject the message
- On reconnection, sequence counters reset to 0

### 2.5 Correlation ID Contract

- Commands carry a UUID correlation ID generated by the sender
- The corresponding Response carries the same correlation ID
- Events are not correlated to commands (they carry their own UUID)
- Both sides must support concurrent in-flight commands (no head-of-line blocking)

### 2.6 Keepalive and Liveness

- Engine sends `EVT_ENGINE_HEARTBEAT` every 5 seconds
- If the API layer receives no messages for 15 seconds, it considers the engine dead
- API layer reconnects with exponential backoff: 1s, 2s, 4s, 8s, 16s, max 30s
- On reconnection, the API layer sends `CMD_GET_ENGINE_STATUS` to synchronize state

## 4. State Machine Contract

Both TypeScript and C++ MUST implement the identical position state machine. Any implementation that allows a transition not in this table is non-conforming.

### 4.1 States (13 total)

| State | Numeric Value | Terminal | Capital at Risk |
|-------|---------------|----------|-----------------|
| INTENT | 0 | No | No |
| PRIMARY_SUBMITTING | 1 | No | Yes |
| PRIMARY_FILLED | 2 | No | Yes |
| HEDGE_SUBMITTING | 3 | No | Yes |
| ACTIVE | 4 | No | Yes |
| CLOSE_TRIGGERED | 5 | No | Yes |
| CLOSING | 6 | No | Yes |
| PRIMARY_CLOSED | 7 | No | Yes |
| COMPLETED | 8 | Yes | No |
| ROLLING_BACK | 9 | No | Yes |
| ROLLED_BACK | 10 | Yes | No |
| ERROR | 11 | No* | Yes |
| CANCELLED | 12 | Yes | No |

*ERROR allows recovery transitions (operator-initiated).

### 4.2 Transition Table

```
INTENT             -> PRIMARY_SUBMITTING, CANCELLED, ERROR
PRIMARY_SUBMITTING -> PRIMARY_FILLED, ERROR, CANCELLED
PRIMARY_FILLED     -> HEDGE_SUBMITTING, ROLLING_BACK, ERROR
HEDGE_SUBMITTING   -> ACTIVE, ROLLING_BACK, ERROR
ACTIVE             -> CLOSE_TRIGGERED, CLOSING, ERROR
CLOSE_TRIGGERED    -> CLOSING, ERROR
CLOSING            -> PRIMARY_CLOSED, COMPLETED, ERROR
PRIMARY_CLOSED     -> COMPLETED, ERROR
COMPLETED          -> (terminal)
ROLLING_BACK       -> ROLLED_BACK, ERROR
ROLLED_BACK        -> (terminal)
ERROR              -> ROLLING_BACK, CLOSING, COMPLETED
CANCELLED          -> (terminal)
```

### 4.3 State Machine Rules

1. Every transition MUST be logged to WAL (PostgreSQL) BEFORE the exchange call
2. The WAL entry status follows: PENDING -> EXECUTING -> CONFIRMED | FAILED | SUPERSEDED
3. On crash recovery, uncommitted WAL entries are reconciled with actual exchange state
4. The `EvtPositionStateChanged` event MUST be emitted for every transition
5. The `exchange_verified` field in the event indicates whether the transition was confirmed by querying the exchange (not just assumed from API response)

### 4.4 Numeric Encoding

The `PositionStateEnum` in `events.fbs` assigns stable numeric values to each state. These values are part of the protocol contract and must never be reassigned. New states are added at the end with the next available number.

## 5. Shared Memory Ring Buffer

### 5.1 Purpose

High-frequency market data (price ticks) flows through a lock-free SPSC ring buffer in shared memory. This avoids UDS serialization overhead for the hottest data path.

### 5.2 Memory Layout

Total size: `128 + (slot_count * 64)` bytes.

#### Header (128 bytes at offset 0)

| Offset | Size | Type | Field | Description |
|--------|------|------|-------|-------------|
| 0 | 4 | uint32 | magic | 0x424F5442 ("BOTB") |
| 4 | 4 | uint32 | version | Protocol version (1) |
| 8 | 4 | uint32 | slot_count | Must be power of 2 |
| 12 | 4 | uint32 | slot_size_bytes | Must be 64 |
| 16 | 8 | uint64 | write_cursor | Monotonic, updated after each write |
| 24 | 8 | uint64 | read_cursor | Advisory, updated by reader |
| 32 | 8 | uint64 | created_at_ns | Monotonic clock at initialization |
| 40 | 4 | uint32 | writer_pid | PID of writing process |
| 44 | 4 | uint32 | reader_pid | PID of reading process (advisory) |
| 48 | 80 | bytes | reserved | Must be zero |

#### Slot (64 bytes, cache-line aligned)

| Offset | Size | Type | Field | Description |
|--------|------|------|-------|-------------|
| 0 | 8 | uint64 | sequence | Monotonic, written LAST by writer |
| 8 | 8 | uint64 | timestamp_ns | Monotonic clock when tick was received |
| 16 | 8 | float64 | bid_price | Best bid price |
| 24 | 8 | float64 | ask_price | Best ask price |
| 32 | 8 | float64 | bid_qty | Depth at best bid |
| 40 | 8 | float64 | ask_qty | Depth at best ask |
| 48 | 2 | uint16 | exchange_id | ExchangeId enum value |
| 50 | 2 | uint16 | symbol_id | Index into symbol lookup table |
| 52 | 4 | uint32 | flags | Bit 0: stale, Bit 1: exchange disconnected |
| 56 | 8 | uint64 | checksum | XOR of bytes [0..55] as 7 uint64 LE values |

### 5.3 Lock-Free SPSC Protocol

**Writer** (one process, one thread):
1. Calculate `slot_index = write_cursor & (slot_count - 1)`
2. Write all fields except `sequence`
3. Compute checksum
4. Write `checksum`
5. Store `sequence = write_cursor + 1` with release semantics
6. Store `write_cursor` in header with release semantics

**Reader** (one process, one thread):
1. Load `write_cursor` from header with acquire semantics
2. If `read_cursor >= write_cursor`, no new data (return empty)
3. If `write_cursor - read_cursor > slot_count`, overrun detected; advance `read_cursor`
4. Calculate `slot_index = read_cursor & (slot_count - 1)`
5. Load `sequence` with acquire semantics -> `seq1`
6. If `seq1 < read_cursor + 1`, slot not yet written (return empty)
7. Read all data fields
8. Load `sequence` again -> `seq2`
9. If `seq1 != seq2`, torn read (writer overwrote during read); retry up to 8 times
10. Verify checksum; if mismatch, report corruption
11. Advance `read_cursor`; store advisory `read_cursor` in header

**Memory ordering (C++ equivalent)**:
- Writer: all data writes use relaxed stores; `sequence` write uses `std::memory_order_release`
- Reader: `sequence` reads use `std::memory_order_acquire`; data reads use relaxed loads
- TypeScript: uses `Atomics.store` / `Atomics.load` on `BigUint64Array` for the sequence field

### 5.4 Overflow Handling

The ring buffer is designed to lose old data rather than block the writer:

- Writer NEVER waits; it always overwrites the oldest slot
- Reader detects overrun by comparing cursor distances
- On overrun, reader logs the gap and jumps forward to the oldest available slot
- Overrun frequency should be monitored; sustained overrun means the reader is too slow

### 5.5 Shared Memory File

- Created by the writer process at the path specified by `MARKET_DATA_SHM_PATH` env var
- Default path: `/tmp/bot-market-data.shm`
- The reader opens the same file and maps it into its address space
- On Linux: use `/dev/shm/` for true shared memory (no disk I/O)
- On macOS (development): use `/tmp/` (file-backed mmap is fine for development)

### 5.6 Symbol Lookup Table

Ring buffer slots use `uint16 symbol_id` instead of strings for compactness. Both processes must share the same symbol-to-ID mapping:

- The writer creates the mapping at startup and writes it to a JSON sidecar file at `{shm_path}.symbols.json`
- The reader loads this file before opening the ring buffer
- Format: `[{"symbol": "BTCUSDT", "id": 0}, {"symbol": "ETHUSDT", "id": 1}, ...]`
- IDs are append-only; never reassigned
- Adding new symbols requires restarting both processes

### 5.7 Recommended Configuration

| Parameter | Development | Production |
|-----------|-------------|------------|
| slot_count | 1024 (64 KB) | 4096 (256 KB) |
| Reader poll interval | 1ms | 100us (busy-poll) |
| Overrun threshold | Log at any | Alert at > 10/sec |

## 6. Error Taxonomy

Hierarchical error codes with retry semantics. Both implementations must use the same codes.

### 6.1 Error Code Hierarchy

```
exchange.
  unreachable         -- Network/DNS failure (retryable after backoff)
  rate_limited        -- 429 response (retryable after delay from header)
  insufficient_balance -- Not enough margin (not retryable)
  invalid_symbol      -- Symbol not found on exchange (not retryable)
  order_rejected      -- Exchange rejected order (context-dependent)
  position_not_found  -- No position for symbol (not retryable)

engine.
  busy                -- Engine not in READY state (retryable)
  shutting_down       -- Engine is shutting down (retryable on reconnect)
  connector_init_failed -- Connector failed to initialize (not retryable)

position.
  invalid_state_transition -- State machine violation (not retryable, bug)
  already_exists      -- Duplicate position ID (not retryable)
  not_found           -- Position ID unknown (not retryable)
  leg_failed          -- One leg of the pair failed (handled by rollback)
  rollback_failed     -- CRITICAL: rollback itself failed (escalate)

validation.
  failed              -- Input validation failed (not retryable, fix input)
  missing_credentials -- No API keys for exchange (not retryable)

internal.
  wal_write_failed    -- WAL write to PostgreSQL failed (retryable)
  db_error            -- General database error (retryable)
  unexpected          -- Catch-all (not retryable, bug)
```

### 6.2 Retry Semantics

| Category | Retryable | Strategy |
|----------|-----------|----------|
| `exchange.unreachable` | Yes | Exponential backoff: 1s, 2s, 4s, max 30s |
| `exchange.rate_limited` | Yes | Respect `Retry-After` header; fallback 60s |
| `engine.busy` | Yes | Linear backoff: 500ms intervals, max 5 retries |
| `engine.shutting_down` | Yes | Wait for reconnect, then retry once |
| `internal.wal_write_failed` | Yes | 3 retries with 100ms intervals |
| All others | No | Report error to caller immediately |

### 6.3 Critical Errors (Operator Escalation)

These errors require human intervention:

- `position.rollback_failed` -- Capital is stuck on exchange(s)
- `exchange.unreachable` during recovery -- Cannot verify exchange state
- State machine violation -- Indicates a bug in the implementation

## 7. Timing Contract

### 7.1 Clock Sources

| Clock | Source | Resolution | Use |
|-------|--------|------------|-----|
| Monotonic | `clock_gettime(CLOCK_MONOTONIC)` / `process.hrtime.bigint()` | Nanoseconds | All `timestamp_ns` fields, latency measurement |
| Wall clock | `gettimeofday()` / `Date.now()` | Milliseconds | WAL `createdAt`/`updatedAt`, human-readable logs |
| Exchange clock | Per-exchange server time API | Milliseconds | Settlement window detection |

### 7.2 Timestamp Authority

- **Ring buffer `timestamp_ns`**: set by the process that receives the WebSocket tick. This is the authoritative "data arrival" time.
- **Envelope `timestamp_ns`**: set by the sender just before writing to UDS. Measures IPC latency when compared to the receiver's clock.
- **`EvtOrderFilled.filled_at_ns`**: set when the fill confirmation arrives from the exchange WebSocket/REST response. This is the authoritative "fill time".
- **WAL timestamps**: wall-clock `Date` objects for PostgreSQL compatibility.

### 7.3 Clock Synchronization with Exchanges

The C++ core is responsible for continuous per-exchange clock offset calibration:

1. On connector initialization, perform 5 round-trips to the exchange's server time endpoint
2. Compute offset as: `offset = exchange_time - (request_time + rtt/2)`
3. Use the median of 5 samples (discard outliers > 2 standard deviations)
4. Re-calibrate every 60 seconds with a single probe
5. Report offsets in `EvtEngineHeartbeat.clock_offsets`
6. Use calibrated offset for settlement window countdown (`EvtCountdownTick.clock_offset_us`)

### 7.4 Latency Budget

Target latencies for the order submission pipeline:

| Step | Target (p50) | Target (p99) |
|------|-------------|-------------|
| Market data: exchange WS -> ring buffer slot | < 50us | < 200us |
| Ring buffer slot -> reader process | < 10us | < 50us |
| IPC command: API -> Engine (UDS) | < 200us | < 1ms |
| Pre-trade slippage check | < 5us | < 20us |
| Order submission: Engine -> Exchange API | < 10ms | < 50ms |
| Fill confirmation: Exchange -> Engine | < 50ms | < 200ms |
| End-to-end: signal -> both legs filled | < 100ms | < 500ms |

## 8. C++ Core Scope Definition

### 8.1 What the C++ Binary Owns

The C++ execution core is responsible for these performance-critical paths:

#### 8.1.1 Time Synchronization
- Continuous per-exchange clock offset calibration (see Section 7.3)
- NTP-style offset computation with outlier rejection
- Exposes calibrated offsets via heartbeat events
- Used by settlement window detection and countdown timers

#### 8.1.2 Order Submission Pipeline
1. **Pre-trade slippage check**: compare intended price against latest ring buffer tick; reject if slippage exceeds configured threshold
2. **Order submission**: REST API call to exchange with retry logic
3. **Fill confirmation**: poll or listen for fill via WebSocket
4. **Slippage reporting**: calculate actual vs. expected fill price, emit `EvtOrderFilled` with `slippage_bps`
5. **Timeout handling**: if no fill within configured window, cancel and report

#### 8.1.3 Settlement Window Detection
- Observe actual funding payment timestamps (not exchange-reported "next funding time")
- Build a statistical model of observed settlement windows per exchange/symbol
- Detect settlement start based on observed behavior, not countdown timers
- Emit `EvtCountdownTick` with corrected countdown based on observed offsets

#### 8.1.4 Position State Tracking
- Mirrors the TypeScript state machine EXACTLY (same 13 states, same transitions)
- Writes WAL entries to PostgreSQL via a minimal libpq interface
- On state transitions, emits `EvtPositionStateChanged` over UDS
- The C++ state machine is the authoritative source when running in native mode

#### 8.1.5 Fill Confirmation and Slippage Guard
- After order submission, confirm fill via REST API query (not just API response)
- If slippage exceeds the configured per-symbol threshold, trigger emergency close
- Log all fill details with nanosecond timestamps for post-trade analysis

#### 8.1.6 Safe Abort / Emergency Close
- Triggered by: slippage guard, liquidation proximity, operator command, signal handler
- Closes all legs as fast as possible using reduce-only market orders
- If one leg fails to close, retries 3 times with 500ms intervals
- If still failing, emits ERROR state and escalates to operator
- SIGTERM/SIGINT handlers trigger safe abort before process exit

### 8.2 What the C++ Binary Does NOT Own

| Responsibility | Owner | Reason |
|----------------|-------|--------|
| Strategy selection | TypeScript API layer | Business logic changes frequently; interpreted language is more agile |
| Risk configuration | TypeScript API layer + PostgreSQL | User-facing settings, not latency-critical |
| UI/Dashboard | Angular frontend | Completely separate concern |
| Analytics/Reporting | TypeScript API layer | Batch processing, not real-time |
| Billing/Credits | TypeScript API layer | Business logic |
| Authentication/Authorization | TypeScript API layer | Security boundary stays in the API layer |
| Database schema migrations | Prisma | ORM handles schema evolution |
| Exchange connector selection | TypedConnectorFactory (TypeScript) | Routing logic; C++ registers its connectors but TS decides which to use |

### 8.3 C++ Build and Deployment

- Single static binary (no shared library dependencies except libc, libpq, libssl)
- Communicates exclusively via UDS and shared memory (no network listeners)
- Started by the TypeScript engine process via `child_process.spawn()`
- Shares the same `EXECUTION_ENGINE_SOCKET_PATH` and `MARKET_DATA_SHM_PATH`
- Graceful shutdown via SIGTERM (30-second timeout, then SIGKILL)
- Crash handling: TypeScript engine detects child exit and runs WAL recovery

## 9. Migration Bridge Design

### 9.1 ConnectorMode Routing

The `TypedConnectorFactory` supports three modes per exchange:

| Mode | Primary | Shadow | Description |
|------|---------|--------|-------------|
| `typescript` | TypeScript | None | Current behavior (Phase 1) |
| `native` | C++ | None | Full C++ execution |
| `shadow` | TypeScript | C++ | Both run; TS is authoritative; divergence logged |

Mode is configured per-exchange via environment variable or database setting:

```
CONNECTOR_MODE_BYBIT=shadow
CONNECTOR_MODE_BINANCE=typescript
CONNECTOR_MODE_BINGX=typescript
```

### 9.2 Fallback Behavior

When `native` mode is configured but the C++ binary is unavailable:

1. `TypedConnectorFactory.create()` checks `nativeRegistry` for the exchange
2. If not registered, logs a warning and falls back to `typescript` mode
3. Emits a `connector_mode_fallback` metric for monitoring
4. Does NOT silently degrade -- the log message is explicit

This is already implemented in `typed-connector-interface.ts` (see `TypedConnectorFactory.create()`).

### 9.3 Shadow Mode Operation

Shadow mode (implemented in `connector-bridge.ts`) runs both implementations in parallel:

**For read-only operations** (getBalance, getPosition, getMarketPrice):
- Both connectors execute in parallel via `Promise.allSettled()`
- TypeScript result is always returned to the caller
- Results are compared; divergences are logged with field, TS value, native value, and relative difference
- Configurable divergence threshold (default: 0.1% for prices, exact match for enums)

**For mutating operations** (placeMarketOrder, closePosition):
- Only the primary (TypeScript) connector executes the actual order
- Native connector does NOT submit orders (safety guarantee)
- Timing comparison is still logged (how fast would native have been?)

**Shadow comparison results** are delivered via callback for storage/analysis:
```typescript
interface ShadowComparisonResult {
  exchange: string;
  operation: string;
  symbol?: string;
  typescriptLatencyUs: number;
  nativeLatencyUs: number;
  divergence: ShadowDivergence | null;
  timestamp: Date;
}
```

### 9.4 Migration Path

Exchanges are migrated to C++ in this order:

#### Phase 2a: Bybit (First)

**Why Bybit first:**
1. Bybit has the best-documented V5 API with consistent response formats
2. Bybit is used by the Bybit-specific funding strategy service (most latency-sensitive)
3. The existing `BybitWithCacheConnector` already handles response normalization, providing a reference implementation
4. Bybit supports both futures and spot in a single API, reducing connector complexity
5. The team has the most operational experience with Bybit's behavior (settlement timing, rate limits)

**Migration steps for Bybit:**
1. Implement C++ Bybit connector satisfying `ITypedExchangeConnector` interface
2. Register via `TypedConnectorFactory.registerNative('BYBIT', factory)`
3. Deploy in `shadow` mode for 1 week minimum
4. Analyze shadow comparison logs for divergences
5. If < 0.01% divergence rate across 10,000+ operations, switch to `native`
6. Keep TypeScript connector warm (initialized but idle) for 1 additional week
7. Remove fallback after confidence period

#### Phase 2b: Binance

Second priority due to volume and the Binance Futures connector already being well-tested.

#### Phase 2c: Remaining exchanges

BingX, Gate.io, KuCoin, MEXC, and CCXT-based exchanges. Lower priority because they contribute less volume and have more API quirks.

### 9.5 Rollback Procedure

If the C++ connector shows issues in production:

1. Set mode to `typescript` via environment variable or admin API
2. Restart the execution engine (picks up new mode)
3. The TypeScript connector resumes as primary immediately
4. No position state is lost (WAL is in PostgreSQL, not in-process)
5. In-flight orders on the C++ side are detected during WAL recovery

## 10. Protocol Versioning

### 10.1 Version Field

The `EvtEngineReady` event includes a `protocol_version` field (uint32). Both sides must agree on the protocol version.

### 10.2 Compatibility Rules

- **Patch version** (version % 100): new optional fields added to existing tables; backwards compatible
- **Minor version** (version / 100 % 100): new message types added; old messages unchanged
- **Major version** (version / 10000): breaking changes to existing message layouts or state machine

Current version: **10001** (major=1, minor=0, patch=1)

### 10.3 Version Negotiation

1. Engine sends `EvtEngineReady` with its `protocol_version`
2. API layer checks compatibility
3. If incompatible, API layer logs error and refuses to send commands
4. No automatic downgrade (explicit version match required)

## 11. Testing Contract

Both implementations must pass these verification tests:

### 11.1 State Machine Conformance

- Every valid transition in the table must succeed
- Every invalid transition must be rejected
- Terminal states must reject all transitions (except ERROR recovery)
- All 13 states must be reachable from INTENT via some sequence of transitions

### 11.2 Ring Buffer Correctness

- Single-threaded: write N ticks, read N ticks, verify all match
- Overrun: write 2 * slot_count ticks, verify reader detects overrun and recovers
- Torn read simulation: verify checksum detects corruption
- Concurrent: writer and reader in separate threads, verify no data corruption over 1M ticks

### 11.3 IPC Round-Trip

- Send each command type, verify response is received with correct correlation ID
- Send command during shutdown, verify REJECTED response
- Verify heartbeats arrive at configured interval (within 10% tolerance)
- Verify sequence numbers are monotonic across 1000 messages

### 11.4 Timing Accuracy

- Verify monotonic timestamps never go backwards
- Verify round-trip measurement is within 10% of wall-clock measurement
- Verify exchange clock offset is within 100ms of manual NTP measurement
